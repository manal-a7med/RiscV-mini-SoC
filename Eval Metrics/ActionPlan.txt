    The Critical Fix: Byte vs. Word Addressing
Look at your program.hex:
    The first line is @00000000.
    The next address tag is @0000000D.
    The one after that is @00000012.

The Problem: objcopy is using byte addresses (hex D is 13, hex 12 is 18). However, your Verilog memory reg [31:0] mem [0:1023] uses word indices. When Verilog's $readmemh sees @0000000D, it will try to put that data into the 13th 32-bit word (which is byte address 52), leaving a huge gap of zeros in your memory. Your CPU will fetch these zeros, execute a "NOP" or illegal instruction, and crash.
The Solution:
You need to tell objcopy to ignore the address tags or use a byte-to-word translation. The easiest way for a student project is to strip the addresses and let Verilog load them sequentially.
'$(BUILD_DIR)/program.hex: $(BUILD_DIR)/program.elf
	$(OBJCOPY) -O verilog --verilog-data-width 4 $< $@
        sed -i '/@/d' $@ '
(The sed command deletes all lines starting with @, ensuring the data is loaded continuously into your memory array).

This is the most robust way to handle address decoding in a VLSI project. By using range-based checks (Base to Base + Offset), you avoid the "aliasing" problems that happen with bit-slicing.
In a VLSI implementation, this "Comparator" style logic is very clean for the synthesis tool to map into standard cells, and it's easy to explain to your professor.

About that xxxxxxxx at 00010ffc
You saw TIME: 630000 | ADDR: 00010ffc | DATA: xxxxxxxx. This is your Stack Pointer initialization. The CPU tried to read from the very top of your Data Memory (0x10000 + 0x0FFF), but since that memory was never initialized with $readmemh, it returned X.* This is normal for a cold boot, as long as the CPU writes to the stack before it tries to read from it.*

Most RISC-V C libraries (like a standard putchar implementation) expect a "Ready" or "Empty" bit, which is the inverse of busy.

The C loop likely looks like this:
C
while ((UART_STATUS & 0x1) == 0); // Wait until bit 0 is HIGH (Ready)
UART_DATA = character;
Because your hardware returns 0 when it is ready (not busy), the C code sees that 0 and thinks: "Oh, the UART is busy, I must wait forever." This is why your simulation logs show the CPU hitting address 10000000 (UART) and then looping back to the same instructions over and over.

NDR (Non-Default Rules) are actually the tool trying to help you.
